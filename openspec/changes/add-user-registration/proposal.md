# 提案：添加用户自主注册功能

## 问题陈述

当前系统存在以下问题：

1. **用户创建流程不完整**：管理员可以通过 `POST /api/v1/users` 创建用户，但 `CreateUserRequest` 只包含 `name` 和 `email`，没有密码字段。创建的用户无法通过邮箱和密码登录。

2. **缺少自主注册途径**：普通用户无法自主注册账户，必须由管理员手动创建，这限制了系统的可用性。

3. **初始管理员依赖**：虽然系统支持通过环境变量创建初始管理员，但后续用户管理仍需手动操作。

## 目标

1. 添加用户自主注册功能，允许用户通过邮箱和密码创建账户
2. 新注册的用户默认为 `user` 角色
3. 移除管理员创建普通用户的功能（保留管理员查看和管理用户状态的能力）
4. 确保与现有的 JWT 认证、角色权限控制体系兼容

## 影响范围

### 新增
- 用户注册接口 `POST /api/v1/auth/register`
- `RegisterRequest` 和 `RegisterResponse` 模型
- 注册速率限制（防止恶意批量注册）

### 修改
- `role-based-access` 规范：移除管理员创建用户的场景
- `UserController`：移除 `CreateUser` 方法

### 删除
- `CreateUserRequest` 模型（不再需要）
- `POST /api/v1/users` 接口（管理员创建用户）

## 设计考量

### 注册接口设计
- **路由**：`POST /api/v1/auth/register`（与登录接口在同一认证组下，无需 JWT 鉴权）
- **请求字段**：`name`（必填）、`email`（必填、唯一）、`password`（必填、强度要求）
- **响应字段**：创建的用户信息（不包含密码）
- **默认角色**：`user`
- **默认状态**：`active`

### 密码安全
- 使用 bcrypt 进行密码哈希（cost factor 12，与现有规范一致）
- 密码强度要求：
  - 最少 8 个字符
  - 建议包含大小写字母、数字和特殊字符

### 防滥用措施
- 对注册接口实施速率限制（同一 IP 每小时最多 5 次注册请求）
- 邮箱唯一性检查（返回明确的冲突错误）

### 管理员用户创建
管理员用户只能通过以下方式创建：
1. 初始管理员（环境变量 `INITIAL_ADMIN_EMAIL` 和 `INITIAL_ADMIN_PASSWORD`）
2. 数据库直接操作（运维场景）
3. 将现有普通用户角色升级为 admin（未来可能添加）

## 兼容性

- 与现有的 `jwt-auth` 规范兼容（使用相同的密码哈希机制）
- 与现有的 `role-based-access` 规范兼容（新用户默认为 user 角色）
- 不影响现有的登录、刷新令牌、API Key 管理功能

## 替代方案

### 方案 A：保留管理员创建用户 + 添加密码字段
- 优点：保留管理员对用户的完全控制
- 缺点：增加管理员操作负担，密码管理复杂
- **未采纳**：与自主注册目标冲突

### 方案 B：添加邮件验证流程
- 优点：防止虚假邮箱注册
- 缺点：需要额外基础设施（SMTP 服务），增加复杂度
- **未采纳**：MVP 阶段保持简单，可后续扩展

### 方案 C：开放注册 + 管理员审批
- 优点：平衡开放性和控制
- 缺点：增加用户等待时间，需要审批流程
- **未采纳**：当前阶段采用直接激活

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 恶意批量注册 | 数据库污染、资源耗尽 | IP 速率限制 |
| 弱密码账户 | 安全风险 | 密码强度验证、bcrypt 哈希 |
| 邮箱枚举攻击 | 隐私泄露 | 错误信息模糊处理（统一返回"注册失败"） |
