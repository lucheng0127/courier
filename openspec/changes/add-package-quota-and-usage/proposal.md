# 变更：添加套餐配额和使用统计的后端API接口

## 为什么

平台需要实现套餐功能，以支持对Provider token用量的限制和管理。当前系统已具备基础的使用统计能力，但缺乏：
1. 套餐产品管理（创建、上下架）
2. 用户套餐购买和管理
3. Token配额检测机制
4. 套餐配额使用统计

这些功能对于商业化运营和资源管控至关重要。

## 变更内容

- 新增套餐（Package）数据模型和管理功能
- 新增用户套餐（UserPackage）关联模型
- 新增套餐配额（PackageQuota）和使用记录（PackageUsage）
- 实现套餐管理API（管理员创建、上下架套餐）
- 实现套餐购买API（用户自助购买，含模拟支付）
- 实现配额检测中间件，在Chat API调用前检查剩余配额
- 扩展现有使用统计API，支持按套餐维度统计

## 影响范围

- 受影响的规范：
  - 新增 `package-management` 规范（套餐管理）
  - 新增 `package-purchase` 规范（套餐购买）
  - 新增 `quota-middleware` 规范（配额检测）
  - 扩展 `usage-tracking` 规范（套餐维度统计）

- 受影响的代码：
  - `internal/model/` - 新增套餐相关模型
  - `internal/controller/` - 新增套餐管理和购买控制器
  - `internal/service/` - 新增套餐服务
  - `internal/repository/` - 新增套餐数据访问层
  - `internal/middleware/` - 新增配额检测中间件
  - `internal/migrate/` - 新增数据库表

## 设计决策摘要

1. **配额计算方式**：按请求扣减，每次请求后直接从套餐配额中扣减实际使用的token数
2. **有效期限制**：套餐购买后有有效期（如30天、90天、365天），到期后配额清零
3. **购买方式**：用户自助购买，包含模拟支付（用户点击购买后直接获得套餐）
4. **扣减策略**：按过期时间优先扣减即将过期的套餐配额
5. **套餐不可修改**：套餐创建后不允许修改，保证已购买用户的权益不受影响
6. **套餐可下架**：下架后不能被购买，但已购买的套餐继续有效

## 数据库变更

新增表：
- `packages` - 套餐产品表
- `package_quotas` - 套餐配额表（一个套餐可包含多个provider的配额）
- `user_packages` - 用户套餐关联表
- `package_usage_records` - 套餐使用记录表
